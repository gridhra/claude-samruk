# /research フェーズ詳細

## Phase 0 — 計画（司令塔）

1. 問いをサブクエスチョンに分解する（並列化できるよう、互いに独立になるように切る）。
2. `knowledge_base/_index_lite.md` と `research_notes/INDEX.md` を確認し、既知の知見を計画から除外する。
3. サブクエスチョン数から複雑度を判定し（orchestration 規則の表）、ワーカー編成を決める。
4. 計画（サブクエスチョン一覧・ワーカー編成・ソース定義の要否）を短くユーザーに提示してから進む。

## Phase 1 — ソース定義

1. `knowledge_base/sources/REGISTRY.md` で対象領域の定義の有無を確認する。
2. 無ければ `/sources` の手続きに従い作成する（researcher ワーカーに委譲してよい）。
3. 以後の全ワーカーに、ソース定義ファイルのパスを渡して従わせる。

## Phase 2 — 並列調査ファンアウト

- researcher を並列起動する（1バッチ最大5）。各 researcher への指示に必ず含めるもの:
  1. 担当サブクエスチョン（1ワーカーにつき1個）
  2. ソース定義ファイルのパス（最初に読ませる）
  3. 返却形式: `source-quality.md` の構造化サマリ（結論・典拠・A〜E評価・確信度・限界）
  4. 生の全文・長い引用を返さないこと
- 収集結果が大量のリスト・URL群になった場合のみ、scout（haiku）に重複除去・分類を分割入力で委ねる。
- ワーカーが返した構造化サマリのうちファイル化されていないものは、**司令塔が `research_notes/` に転記保存**し INDEX に追記する（Phase 3 で synthesizer が全サマリをファイルから読めるようにするため）。ワーカーが保存を報告した場合もファイルの実在を確認する。

## Phase 3 — 統合・起草（synthesizer）

- synthesizer に全サマリと保存先パス（`research_notes/YYYY-MM-DD-<slug>.md`）を渡し、以下をさせる:
  1. 矛盾するサマリの明示と調停
  2. `templates/report_full.md` の構造での草稿起草（**ファイルに直接書かせ**、司令塔へはパス・要旨・「網羅性の穴」のみ返させる）
  3. 追加調査が必要な点の列挙
- 穴が重大なら Phase 2 に戻り、追加の researcher で埋めてから再統合する。

## Phase 4 — 反証探索

- 草稿の主要結論を列挙し、それぞれへの反対仮説・反証ソースを researcher に探させる（反証専用の指示とする）。
- 反証が見つかった場合: synthesizer に草稿を修正させる（結論の修正、または反証への応答の追記）。
- 見つからなかった場合も、その旨と探索範囲を報告書の「反証・対立見解」節に記す。

## Phase 5 — 評価（verifier）

- verifier に草稿ファイルのパスを渡し、5軸評価をさせる:
  - 引用の抜き取り検証（主要典拠から数件を実ソースと突き合わせ）を必ず含める
  - 各軸1〜5の採点と根拠
  - 4未満の軸があれば差し戻し先フェーズの指定
- 不合格の場合、指定フェーズへ差し戻す。差し戻し回数は orchestration 規則のサイクル上限まで。
- 上限に達しても未達の場合は、そのまま合格扱いにせず、報告書の「限界・未解決点」節に未達の軸と理由を明記して終える。

## Phase 6 — 永続化（司令塔）

1. 最終報告書の「メタ」節にワーカー数・5軸スコア・差し戻し回数が記載されていることを確認する。
2. `research_notes/INDEX.md` に1行追記する（knowledge-hygiene 規則のフォーマット）。
3. 確立知識に昇格すべき知見があれば `/archive` の実行をユーザーに提案する（自動昇格しない）。
4. ユーザーへの最終報告は BLUF・ファイルパス・特記事項のみ。報告書全文を会話に再掲しない。
